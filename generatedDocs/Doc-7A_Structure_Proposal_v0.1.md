## Decisions proposed for ratification at structure freeze (R-set)

- **R1 — Doc-7 program shape: a metastandard + surface-partitioned realizations.** Doc-7 = **Doc-7A** (this frontend realization metastandard) + a set of realization documents partitioned **by frontend surface/actor** (Doc-7B…7H below), each authored through the staged-freeze lifecycle and gated at freeze by Appendix A. **The partition axis departs from the Doc-5/Doc-6 backend-module letter map (B=M0…K=M9) by intent:** the UI is not decomposed by backend module — a single route renders content from many modules — so partitioning by deployable UI surface/route-group (mapping to the Doc-5 tri-actor model: Public / User / Admin) keeps each surface document whole. **Cross-cutting shared documents (Doc-7B Design System, Doc-7C App Shell & Data Layer) freeze before the surfaces that consume them.** *(Alternative considered: per-backend-module partition parallel to Doc-5/6. Rejected — it would split a single deployable surface across many documents and break the route-group ownership cadence. Surface partition is the proposed shape; Board ratifies at freeze.)*
- **R2 — Realize-never-redecide; the frozen Doc-5 surface + Doc-4M + Doc-2 are the *what*-authorities.** Every screen binds to a frozen Doc-5x contract **by pointer**; every lifecycle UI binds to a Doc-4M state machine by pointer; every tenancy/permission gate binds to Doc-2 §6/§7 by pointer. Doc-7 coins **nothing** in the domain or the API surface. Physical UI choices (component tree, layout, interaction/visual design, optimistic-update strategy) are the only *new* decisions, and they MUST NOT change domain meaning or invent a contract/state/permission. Gaps → `[ESC-7-*]` markers, escalated to the owning document, never invented.
- **R3 — Stack realization = the fixed Master-Architecture frontend stack.** Realize the already-decided stack (Master Architecture / CLAUDE.md §2): **Next.js 15 App Router + React + Tailwind + shadcn/ui** (exact version pins live in `package.json` once code exists — CLAUDE.md §2). Server Components are the default; client components only where interaction requires; **server actions / route handlers are the write path** to the frozen Doc-5 contracts. The component-kit boundary (shadcn/ui primitives → app components → surface composition) is realized in Doc-7B. No stack element is coined or re-decided.
- **R4 — Consistency with the frozen Doc-5 API surface (not conformance to Doc-5A).** Every screen's reads/writes/lists MUST map to a frozen `Doc-5x` contract: list views consume the cursor-paginated contracts (`Doc-5A §8`) bounded by the contract's `*.list_page_size_max` POLICY key (Doc-3 §12, by pointer — never a literal); mutations call the frozen command contracts honoring `*.idempotency_dedup_window`; error rendering maps the `Doc-5A §6.2` error taxonomy to user-facing states (no invented error). The typed API-client (Doc-7C) binds to the frozen contracts. **Doc-5A has no conformance authority over Doc-7** (`Doc-5 Governance Note §8.5`) — surfaced as `[ESC-7-API]` only where a screen needs data no frozen Doc-5 contract provides (flag-and-halt; Board → additive Doc-5x patch — never resolved locally).
- **R5 — Content ≠ Presentation (Invariant #9 / Golden Rule #4) — the load-bearing frontend invariant.** The frontend strictly separates authoritative **content** (module-owned, fetched via contracts) from **presentation** (layout, theme, microsite skinning, display ordering). Presentation never mutates, caches-as-authoritative, or stands in for content; a vendor microsite's presentation layer renders M2 content but owns no authoritative data (the same M2 microsite content is rendered read-only by the Public surface and managed by the Vendor workspace — two surfaces, one owner). **Display sort/filter is presentation over the contract-provided result set, never a re-ranking of governed matching** — the matching/sorting authority is M3 (`Doc-5E`); the UI displays results, never re-scores them. Realized via the design-system/presentation-component boundary (Doc-7B) + per-surface composition.
- **R6 — Users Act, Organizations Own (Invariant #5) — active-org context realization.** Every authenticated surface resolves the **active organization context server-side**; the client-supplied org ID is **never trusted** (CLAUDE.md §5). Org switching, membership/role-scoped navigation, and every tenant-scoped read/write carry the server-validated active-org context. Realized in the App Shell (Doc-7C) as the server-resolved org-context boundary; surfaces consume it, never re-derive it client-side. **A surface is a route-group/capability, not a per-user exclusive app:** one user under one active org may mount Buyer **and** Vendor workspaces simultaneously where Platform Participation is Hybrid (Invariant #2); the App Shell composes the navigable surface set from the user's platform participation + org role.
- **R7 — Authorization is app-layer; the UI gate is UX, not the model (CLAUDE.md §2).** The frontend gates navigation/actions on Doc-2 §7 permission slugs (bound by pointer) for **UX only** — hide/disable what the user may not do — but **the server re-validates every action**; UI gating is never the authorization boundary. The UI reads **entitlements (M7) and permissions (M1) via contracts**, never plan-name or role-name string checks (Invariant #10; M7 entitlement discipline — `Doc-5I` R10). No permission slug or entitlement key is coined.
- **R8 — Private Exclusion Stays Private (Invariant #11) — non-disclosure in the UI.** A blacklisted/excluded vendor's frontend experience is **byte-equivalent** to a non-matched vendor's: no surface, view, count, analytic, notification, or error reveals buyer-private exclusion (`buyer_vendor_statuses`) or link suggestions. The buyer-private CRM (M4) renders **only inside the buyer's own workspace** and never leaks to any vendor-facing surface. Realized as a per-surface non-disclosure obligation (Doc-7F/7G) + an Appendix A check; the Vendor workspace (Doc-7G) carries the load-bearing byte-equivalence attestation.
- **R9 — State-machine-driven UI (Doc-4M authority).** Every lifecycle surface (RFQ, quotation, verification, subscription, post-award documents) renders state and available transitions **strictly from the Doc-4M state machine**: the UI offers only transitions the machine permits for the actor in the current state; it never invents a transition or a state label. Optimistic UI is permitted but **reconciles to the server's authoritative state**; the UI displays state, the server owns it. No state or edge is coined.
- **R10 — AI Suggests, UI Surfaces, Modules Decide (Invariant #12).** AI-derived artifacts (M9 — `Doc-5K`) render **only as clearly-labeled advisory surfaces** (suggestions, drafts, summaries); they never auto-commit a business decision and never display as authoritative. The regenerable/TTL nature of `ai.*` reads (`Doc-5K` R7) is honored — AI panels are disposable projections, never a UI source of truth. **No AI surface gates a procurement or governance decision.**
- **R11 — Accessibility, i18n, currency & performance baseline.** Realize a baseline every surface inherits (from Doc-7B): a WCAG-AA target with keyboard/focus semantics; **multi-currency display per value field, default BDT, currency never assumed** (Doc-2 §0.4); Bengali/English i18n-readiness (localized copy is presentation; authoritative data is module-owned); responsive breakpoints; and the App Router performance contract (RSC streaming, suspense boundaries, image/font optimization). These are realization conventions and coin no domain element.
- **R12 — Out-of-frontend boundary (the presentation analog of the out-of-wire / out-of-DB fence).** The frontend owns **no authoritative state** — only ephemeral view/interaction state; all truth is fetched via Doc-5 contracts and persisted server-side. File uploads go to **Supabase Storage** via the contract-mediated path (the UI holds a `file_ref`, never authoritative blobs); **Supabase Realtime** is a transport for live updates, not a store; client-side data caches (e.g. query-cache libraries) are **disposable projections, never source of truth**. **Flag-and-halt if any UI component is proposed as the authoritative owner of business state.**
