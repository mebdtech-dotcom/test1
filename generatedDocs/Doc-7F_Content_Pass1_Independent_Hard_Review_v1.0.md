### MAJOR-1 — §2 implies the buyer invites vendors from discovery; **the buyer has no invite command** (invitations are engine-generated — the moat)
§2.1 frames discovery as "vendor discovery **to invite**." **Verified `Doc-5E`:** there is **no buyer invite / send_invitation / target command** — only `respond_to_invitation` (Vendor) and invitation **reads** (`get_invitation`/`list_invitations`); **invitations are generated by the engine** (`assemble_and_route_wave`, §8 **System out-of-wire**). The buyer **never directly invites a vendor** — that would bypass governed routing (the moat).
**Required fix:** §2 reframe — discovery (`search_catalog`/`list_vendor_directory`/favorites) lets the buyer **research/reference** vendors **while authoring the RFQ** (e.g. as RFQ input the engine considers); the **engine generates invitations out-of-wire per governed routing** (FR3). The buyer **issues no invitation**. If a buyer-targeted-vendor RFQ field exists, it is an **RFQ authoring input** (Doc-5E §4), not a direct invite; if a direct buyer invitation is ever required, it is `[ESC-7-API]` — never assumed/coined.

### MINOR-1 — §3 internal approval is role-gated (approver ≠ submitter)
`submit_rfq` → `approve_rfq`/`reject_internal_rfq` is the org's internal spend authorization; the **approver is typically a different role** (e.g. Director approving an Officer's RFQ).
**Required fix:** §3 note the internal approval is **role-gated** — the approve/reject actions are gated (UX) on the approver's permission slug, distinct from the submitter; server re-validates (`Doc-7A §4.3`).

### MINOR-2 — §2.2 favorites scope
§2.2 says favorites are "own-org / membership-scoped." Pin it: BC-MKT-7 is **membership-only** — favorites belong to the **member's context within the org**.
**Required fix:** §2.2 state favorites are **per-membership** (the user's membership in the active org), consistent with BC-MKT-7 (no slug, membership-only).

### NITPICK-1 — §3.3 confirm the Doc-4M routed/published state label
§3.3 writes "(routed by the engine)" as an RFQ state. The actual state label is Doc-4M's, not to be invented.
**Required fix:** bind the post-approval RFQ state to its **Doc-4M label** by pointer (confirm at content); don't coin "routed."
